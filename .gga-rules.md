# AlbumSL GGA Rules

## Objetivo

Revisar diffs de AlbumSL con foco en bugs reales, seguridad, arquitectura y build, gastando la menor cantidad posible de tokens.

No repetir reglas completas del proyecto. No hacer review filosófica. No comentar estilo menor salvo que cause un problema real.

## Prioridad Critical

Reportar como Critical si el diff:

- Expone secretos reales, tokens, private keys o credenciales.
- Permite escritura sensible desde frontend.
- Permite que el cliente asigne figuritas, abra sobres o modifique inventario.
- Desactiva autenticación o validación de ownership.
- Relaja Firestore Rules de forma insegura.
- Rompe aislamiento de usuarios.
- Deploya o habilita producción sin aprobación explícita.

## Prioridad High

Reportar como High si el diff:

- Hace que `packages/domain` importe Firebase, React, functions o infraestructura.
- Hace que `packages/application` importe Firebase, React, functions o infraestructura.
- Mueve lógica de negocio sensible a `apps/web`.
- Confía en `uid`, email, role o claims enviados desde el body del cliente.
- Escribe `packOpenings`, `packClaims`, `userStickers.quantity` o contadores sensibles desde cliente.
- Agrega una feature sensible sin tests relevantes.
- Rompe TypeScript estricto o introduce `any` sin justificación.
- Rompe build, typecheck, lint o tests.
- Agrega dependencias innecesarias o riesgosas.

## Prioridad Medium

Reportar como Medium si el diff:

- Duplica reglas de dominio en frontend.
- Mezcla responsabilidades de capas.
- Agrega queries Firestore directas en componentes grandes.
- No maneja estados loading/error/empty en una pantalla nueva.
- No documenta una decisión técnica relevante.
- Deja errores internos o stack traces visibles al usuario.
- Hace tests frágiles por timestamps, random no inyectado o dependencia de infraestructura real.

## Prioridad Low

Reportar como Low solo si aporta valor claro:

- Naming confuso que puede causar errores.
- Documentación desactualizada por el cambio.
- Validación parcial insuficiente para el alcance.
- Pequeñas mejoras de mantenibilidad directamente relacionadas con el diff.

## No comentar

No comentar:

- Preferencias subjetivas de estilo.
- Refactors grandes fuera del alcance.
- Cambios cosméticos irrelevantes.
- “Podrías mejorar…” sin riesgo concreto.
- Código no tocado por el diff salvo que el cambio lo rompa.
- Reglas generales ya documentadas si no hay violación.

## Formato de respuesta

Responder compacto:

```txt
Critical
- archivo:línea — problema — fix sugerido

High
- archivo:línea — problema — fix sugerido

Medium
- archivo:línea — problema — fix sugerido

Low
- archivo:línea — problema — fix sugerido

Summary
- Aprobado / No aprobado
- Riesgo principal
- Validación recomendada
```

Si no hay hallazgos relevantes:

```txt
Summary
- Aprobado.
- No encontré riesgos relevantes en el diff.
- Validación recomendada: <comando mínimo aplicable>.
```

## Reglas específicas AlbumSL

- Firebase es infraestructura, no dominio.
- `packages/domain` debe mantenerse puro.
- `packages/application` debe depender de puertos/interfaces.
- `packages/infra-firebase` puede acoplarse a Firebase.
- `functions` y `apps/api` deben ser adapters/server-side.
- `apps/web` no debe contener lógica sensible del álbum.
- Firestore Rules deben negar escrituras sensibles por defecto.
- Dev actual es Spark-only.
- No deployar Cloud Functions reales en Firebase Spark.
- No tocar prod sin aprobación explícita.
